# DataGuard Client — Десктопный клиент

## Обзор проекта

**DataGuard Client** — десктопное приложение с графическим интерфейсом, разработанное на **Avalonia UI** для кроссплатформенной работы. Приложение является клиентской частью системы DataGuard и предоставляет пользовательский интерфейс для взаимодействия с backend API (ASP.NET Core) через gRPC/REST.

## Ключевые компоненты

### ViewModelBase
Базовый класс для всех ViewModel, наследует `CommunityToolkit.Mvvm.ComponentModel.ObservableObject`. Обеспечивает INotifyPropertyChanged и поддержку `[ObservableProperty]`.

### SidebarViewModel
Управляет навигацией:
- `TopItems` / `BottomItems` — коллекции пунктов меню
- `SelectedItem` — выбранный пункт
- `IsExpanded` — состояние боковой панели
- `ToggleExpandCommand` — команда сворачивания/разворачивания
- `SelectItemCommand` — команда выбора пункта

### Sidebar (View)
Пользовательский контрол с:
- Плавной анимацией ширины (через `DispatcherTimer`, 16мс)
- Автоматическим обновлением выделения кнопок
- Переключением видимости текста при сворачивании

### IconLoader
Статический сервис для загрузки SVG-иконок:
- Кэширует результаты в `ConcurrentDictionary`
- Парсит SVG path data в `StreamGeometry`
- Путь: `avares://Client/Assets/icons/{name}.svg`

## Соглашения разработки

- **Nullable**: включён
- **MVVM**: ViewModel → View через `ViewLocator` (автоматическая замена "ViewModel" на "View" в имени типа)
- **Compiled Bindings**: включены (`AvaloniaUseCompiledBindingsByDefault=true`)
- **Валидация**: отключена стандартная `DataAnnotationsValidationPlugin` Avalonia (используется CommunityToolkit)

## Архитектура приложения

### Общая структура: MVVM + Service Layer

```
┌─────────────────────────────────────────────────────────┐
│                      Views (Avalonia)                    │
│   XAML-представления, анимации, привязки, конвертеры     │
└──────────────────────┬──────────────────────────────────┘
                       │  Data Binding (Compiled)
                       ▼
┌─────────────────────────────────────────────────────────┐
│                   ViewModels (MVVM)                      │
│   Состояние UI, команды, валидация, навигация            │
│   CommunityToolkit.Mvvm [ObservableProperty], [RelayCommand]
└──────────────────────┬──────────────────────────────────┘
                       │  Вызов сервисов
                       ▼
┌─────────────────────────────────────────────────────────┐
│                    Services Layer                        │
│   ApiService, AuthService, CacheService, IconLoader      │
│   Асинхронная работа с сетью, БД, файловой системой      │
└──────────────────────┬──────────────────────────────────┘
                       │  HTTP/gRPC вызовы
                       ▼
┌─────────────────────────────────────────────────────────┐
│                Backend (DataGuard Server)                 │
│   ASP.NET Core, gRPC, JWT Auth, PostgreSQL               │
└─────────────────────────────────────────────────────────┘
```

### Поток данных

1. **View** получает данные через `DataContext` (биндинг к ViewModel)
2. **ViewModel** обрабатывает действия пользователя через `[RelayCommand]`
3. **Services** выполняют асинхронные операции (сеть, кэш, валидация)
4. Результаты возвращаются через `ObservableProperty` → UI обновляется автоматически

### Жизненный цикл компонентов

- **App.OnFrameworkInitializationCompleted** — создание MainWindow, установка DataContext
- **ViewLocator** — автоматический поиск View по имени ViewModel (`*ViewModel` → `*View`)
- **OnDataContextChanged** (в code-behind) — подписка на события ViewModel, инициализация анимаций
- **OnDetachedFromVisualTree** — отписка от событий, очистка ресурсов

## Принципы дизайна интерфейса

### Философия: «Невидимый интерфейс»

Интерфейс должен быть **интуитивно понятным** и **не перегруженным**. Пользователь не должен замечать дизайн — он должен решать свою задачу.

### Основные правила

1. **Плоский дизайн (Flat)**
   - Без теней, градиентов, 3D-эффектов
   - Исключение: всплывающие уведомления, модальные окна (допустимо лёгкое наслоение)
   - Контент на одном уровне, визуальная иерархия через размер и цвет

2. **Адаптивные цвета**
   - Использовать системные `{DynamicResource}` Avalonia
   - Поддержка светлой/тёмной темы без хардкода цветов
   - Контрастность для доступности (WCAG AA минимум)

3. **Минимализм**
   - Только необходимая информация на экране
   - Скрытие второстепенных элементов за кнопками/меню
   - whitespace — часть дизайна, не бояться пустого пространства

### Анимации

**Цель:** скрыть резкие переходы, не создавать ощущение заторможенности.

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Длительность | 100–250 мс | Быстро, но заметно |
| Easing | Linear или CubicEaseOut | Без «пружинности» |
| Тип | Fade, Slide (короткие) | Минимальное движение |
| Отключение | Всегда можно отключить | Для слабых ПК |

**Правила:**
- Анимация **не блокирует** UI (использовать `DispatcherTimer` или `Task.Delay`)
- Прерываемая: если пользователь действует во время анимации — она прерывается и идёт к новому состоянию
- Анимировать только `Opacity`, `Width`, `RenderTransform` (GPU-ускоренные свойства)

### Работа с большими данными (Fluent + ANT)

Для страниц с таблицами, списками, формами:

1. **Fluent**: боковая навигация, акриловые фоны, системные цвета
2. **ANT Design**: сетки, карточки, пагинация, фильтры над таблицами

**Приёмы:**
- **Пагинация** или **виртуализация** списков (не грузить всё сразу)
- **Фильтры и поиск** над таблицами
- **Колонки с сортировкой** по клику на заголовок
- **Skeleton-заглушки** вместо пустого экрана при загрузке
- **Контекстные действия** (правый клик или кнопка «⋯» в строке)

### Эргономика и «защита от дурака»

**Цель:** пользователь не может случайно сломать данные или нарушить работу.

1. **Подтверждение деструктивных действий**
   - Удаление, выход, сохранение — диалог «Вы уверены?»
   - Кнопка «Удалить» — красная, в стороне от основных

2. **Валидация на клиенте**
   - Проверка email, длины пароля, обязательных полей **до** отправки
   - Подсветка ошибок inline, не после отправки

3. **Идемпотентность**
   - Двойной клик по кнопке «Сохранить» = один запрос
   - Блокировка кнопки на время выполнения операции

4. **Ограничение прав (клиентская часть)**
   - Скрывать элементы, к которым нет доступа
   - Не показывать «Управление пользователями» если роль ≠ Admin
   - **Важно:** клиентская проверка — UX, серверная — безопасность

### Примеры защиты

| Уязвимость | Клиентская защита | Серверная защита |
|------------|-------------------|------------------|
| SQL-инъекция | Валидация ввода (email, длина) | Параметризованные запросы (EF Core) |
| XSS | Экранирование текста в UI | HTML-encoding на сервере |
| CSRF | Не храните токены в localStorage | SameSite cookies, CSRF-токены |
| Подделка запроса | JWT в памяти, не в файле | Проверка подписи, срок действия |

## Стандарты написания кода (C#)

### Структура файла

```csharp
// 1. Using (системные → сторонние → свои)
using System;
using System.Collections.Generic;
using System.Threading.Tasks;
using CommunityToolkit.Mvvm.ComponentModel;
using Client.Services;

// 2. Namespace
namespace Client.ViewModels
{
    // 3. Класс
    /// <summary>
    /// Краткое описание назначения класса.
    /// </summary>
    public class ExampleService
    {
        // 4. Поля (private, readonly где возможно)
        private readonly HttpClient _httpClient;

        // 5. Конструктор
        /// <summary>
        /// Инициализирует сервис с переданным HTTP-клиентом.
        /// </summary>
        public ExampleService(HttpClient httpClient)
        {
            _httpClient = httpClient ?? throw new ArgumentNullException(nameof(httpClient));
        }

        // 6. Методы (public → private)
        /// <summary>
        /// Загружает данные пользователя с сервера.
        /// Проверяет валидность токена перед запросом.
        /// При ошибке сети возвращает null, не выбрасывает исключение.
        /// </summary>
        public async Task<UserData?> LoadUserDataAsync(string token, CancellationToken ct = default)
        {
            // ... реализация
        }
    }
}
```

### Правила именования

| Элемент | Стиль | Пример |
|---------|-------|--------|
| Класс | PascalCase | `SidebarViewModel` |
| Метод | PascalCase | `LoadDataAsync` |
| Свойство | PascalCase | `IsExpanded` |
| Поле private | `_camelCase` | `_httpClient` |
| Локальная переменная | camelCase | `userData` |
| Параметр | camelCase | `cancellationToken` |
| Константа | PascalCase | `MaxRetries` |
| Interface | I + PascalCase | `IApiService` |

### Асинхронность

**Правила:**
1. Все I/O-операции — `async/await`
2. Суффикс `Async` для асинхронных методов
3. `CancellationToken` передавать во все асинхронные методы (по умолчанию `default`)
4. Не использовать `.Result` или `.Wait()` — только `await`
5. `ConfigureAwait(false)` в сервисном слое (не в ViewModel)

```csharp
public async Task<User?> GetUserAsync(int id, CancellationToken ct = default)
{
    var response = await _httpClient.GetAsync($"/api/users/{id}", ct);
    response.EnsureSuccessStatusCode();
    return await response.Content.ReadFromJsonAsync<User>(ct);
}
```

### Комментарии

**Каждый метод** должен иметь `<summary>` с описанием:
- **Что делает** метод (одна строка)
- **Важные детали** (проверки, побочные эффекты, исключения)
- **Возвращаемое значение** (если не очевидно)

```csharp
/// <summary>
/// Проверяет валидность email по стандарту RFC 5322.
/// Возвращает true если формат корректен, иначе false.
/// Не проверяет существование адреса на сервере.
/// </summary>
public static bool IsValidEmail(string email)
{
    // ...
}
```

**Один файл = один компонент.** Исключения:
- Маленькие DTO-классы можно группировать (до 5 в файле)
- Extension-методы для одного типа — один файл

**Структура папок:**
```

├── Views/           # XAML + code-behind
├── Services/        # Бизнес-логика, API, утилиты
```

### Безопасность кода

1. **Null-проверки**: использовать `?.`, `??`, null-forgiving `!` только когда уверены
2. **Валидация ввода**: проверять все данные от пользователя (длина, формат, диапазон)
3. **Не хардкодить секреты**: токены, ключи — через DI или защищённое хранилище
4. **JWT-токены**: хранить в памяти, не в файлах/реестре
5. **Логирование**: не логировать токены, пароли, персональные данные

### Производительность

**Проблемные места и решения:**

| Операция | Проблема | Решение |
|----------|----------|---------|
| Загрузка списка >100 элементов | Блокировка UI | Виртуализация (`ItemsRepeater`), пагинация |
| SVG-парсинг при каждом рендере | Тормоза при скролле | Кэширование (`IconLoader._cache`) |
| Частые обновления UI | Перерисовка | `ObservableProperty`, батчинг изменений |
| HTTP-запросы без таймаута | Зависание | `CancellationToken`, таймаут 30с |
| Создание объектов в цикле | GC-нагрузка | Object Pool, переиспользование |

**При написании кода указывай в комментарии, если операция может вызвать нагрузку:**

### Мысленная проверка кода

**Перед завершением работы пройдись по каждой строчке и спроси:**
1. Что делает эта строка?
2. Может ли здесь быть `null`?
3. Что если сеть откажет?
4. Что если пользователь введёт некорректные данные?
5. Можно ли сделать быстрее/проще?

**Чеклист:**

#### Правила документации

1. **Каждая публичная функция** — полное описание (что делает, параметры, возвращаемое значение, исключения)
2. **Каждый публичный класс** — описание назначения
3. **Приватные методы** — достаточно `<summary>` с одной строкой
4. **Связи между компонентами** — использовать `<see cref="..."/>` для указания зависимостей
5. **Без воды** — не писать «этот метод делает то, что он делает», только суть

#### Хардкод значений

**Запрещено хардкодить** числовые/строковые значения в логике. Все константы — в переменных или конфигурации.

```csharp
// ❌ Плохо
if (width > 100) { ... }
var items = new List<string> { "home", "settings", "logout" };

// ✅ Хорошо
private const double ExpandedWidth = 250.0;
private const double CollapsedWidth = 60.0;
private static readonly string[] DefaultIconNames = { "home", "settings", "logout" };

if (width > CollapsedWidth) { ... }
```

**Где хранить конфигурацию:**
- UI-константы → `private const` / `private static readonly` в классе
- Строки подключения, URL, таймауты → `appsettings.json` или DI
- Тексты ошибок → ресурсные файлы или константы
